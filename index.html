Before You Start

How to Use This Roadmap

⏱️

The 1-Hour Rule — strictly enforced

Every day has exactly: 1 video (≤15 min) · 1 short read (≤10 min) · 2 hands-on tasks (≤25 min) · 1 problem (≤5 min) · 1 AI prompt to use if stuck (≤5 min). Stop at 1 hour even if you want to do more — consistency over intensity.

01 //

One Day at a Time

Open a day card, do it completely, close your laptop. Do not jump ahead. Each day builds on the one before.

02 //

Code, Don't Watch

The video is a primer — max 15 min. Spend the rest actually writing code. Muscle memory beats passive viewing every time.

03 //

Stuck? Use the AI Prompt

Each day has a purple AI prompt. Paste it into Claude when stuck. Get unblocked fast and move on — don't lose the hour.

04 //

Push Every Project

After each mini project, push to GitHub. By Day 21 you'll have a live portfolio employers can browse. That's the goal.

DAY 03JavaScript — Arrays, Objects & Template Literals1 hr+
Today's Focus
  • Arrays (lists of values), Objects (groups of named values), and Template Literals (modern string building). These three are everywhere in Playwright — your test data, your assertions, and your error messages all use them constantly.
Watch (15 min max)
  • Search: "JavaScript arrays objects template literals beginners" — any video under 15 min. Fireship "JavaScript in 100 Seconds" is a great refresher after watching a longer intro.
Read (10 min)
  • javascript.info/array — read the first section only (Declaration, accessing items). Then read javascript.info/object — the first section (Literals and properties).
Hands-On (30 min)
  • Arrays: const browsers = ['chromium', 'firefox', 'webkit']; → log the first item (browsers[0]) → log the length (browsers.length) → use a for loop to print every item → add a new item with browsers.push('edge'). You'll use arrays constantly for test data.
  • Objects + Template Literals: const user = { username: 'admin', password: 'secret123', role: 'admin' }; → access properties: user.username → use a template literal: console.log(`Logging in as ${user.username}`); — note the backticks. This is exactly how you'll define test data objects for parameterized tests on Day 18.

DAY 15Inspector + page.evaluate() + waitForResponse()1 hr+
Today's Focus
  • Three essential tools used every day in real QA work: Playwright Inspector (step-through debugging, better than trace viewer when writing tests live), page.evaluate() (run JavaScript in the browser — required for date pickers, scroll, canvas, cookies), and page.waitForResponse() (wait for a specific network call — essential for SPAs where navigation doesn't reload the page).
Watch (15 min max)
  • Search: "Playwright Inspector debug mode page.evaluate waitForResponse 2024" — any video showing the Inspector UI in action. Stop at 15 min.
Read (10 min)

Hands-On (30 min)
  • Run any existing failing test with PWDEBUG=1 npx playwright test --headed → the Inspector opens → step through each line using the Step Over button → observe each action highlighted on the page. This is how you write tests live without constantly re-running.
  • Use page.evaluate() to: 1) scroll to the bottom of a page: await page.evaluate(() => window.scrollTo(0, document.body.scrollHeight)), 2) get the inner text of an element that Playwright locators can't reach: const text = await page.evaluate(() => document.querySelector('.tricky-element').innerText). Then write a test on reqres.in using page.waitForResponse('**/api/users') before clicking a button — assert the network response status is 200.

Top 10 Interview Questions — With Answers

Q01What is Playwright and how is it different from Selenium?+

Playwright is a modern automation framework by Microsoft supporting Chromium, Firefox, and WebKit. Unlike Selenium it has built-in auto-waiting, native async/await, network request mocking, a trace viewer, iFrame support via frameLocator, and built-in API testing. It is faster, more reliable, and significantly easier to write and maintain than Selenium for modern web apps.

Q02What is the Page Object Model and why do we use it?+

POM is a design pattern where each page is a class containing that page's locators and actions. Tests use class methods instead of raw selectors. If a locator changes, you fix it in one file — not 20 test files. Code is reusable. Tests are readable. Maintenance cost drops dramatically as the application grows.

Q03How do you handle elements inside iFrames?+

Use page.frameLocator('iframe-selector') to scope all subsequent locators inside the iframe. Example: page.frameLocator('#payment-iframe').getByLabel('Card number').fill('4242...'). For nested iFrames, chain multiple frameLocator calls. Cross-origin iFrames (different domain) cannot be directly accessed due to browser security — you need special handling or the page's cooperation.

Q04How do you test file upload in Playwright?+

Use locator.setInputFiles('path/to/file.txt') on the file input element. For multiple files: setInputFiles(['file1.txt', 'file2.pdf']). For downloads, wrap the click in page.waitForEvent('download') — this resolves with a Download object where you can call download.suggestedFilename() or download.saveAs('path'). No browser dialog handling needed — Playwright intercepts at the protocol level.

Q05How do you run tests across multiple browsers?+

Configure projects in playwright.config.js — one entry per browser using Playwright's built-in devices: Desktop Chrome, Desktop Firefox, Desktop Safari. Run all: npx playwright test. Run one: npx playwright test --project=firefox. The HTML report groups results by browser so you can see exactly which browser failed which test. Add a mobile device project using devices['iPhone 13'] for mobile viewport testing.

Q06What is a custom fixture and how is it better than beforeEach?+

A fixture defined with test.extend() is a reusable, injectable dependency. Unlike beforeEach — which runs for every test in the file whether they need it or not — fixtures are only set up when a test actually requests them by name. They're sharable across multiple test files, composable (fixtures can use other fixtures), and self-documenting. A loggedInPage fixture is cleaner than repeating a beforeEach login in every spec file.

Q07What is storageState and when do you use it?+

storageState saves browser cookies and localStorage to a JSON file after login. All tests load this as their starting state — already authenticated. Configure it in global-setup.js and reference it in playwright.config.js. Benefits: much faster runs (no login per test), no login flakiness, and ability to test multiple roles with separate state files. Add the auth folder to .gitignore so credentials aren't committed.

Q08How do you test APIs in Playwright and what do you assert?+

Use the built-in request fixture: request.get(url), request.post(url, {data:{...}}), request.delete(). Assert: response.status() for HTTP code, response.ok() for 2xx, response.json() for body content. Go beyond status codes — validate schema: check required fields exist, check field types (typeof x === 'string'), check nested objects, check array lengths. Write a reusable schema validator function and call it across multiple tests.

Q09What is a flaky test and how do you fix it?+

A flaky test passes sometimes and fails sometimes without code changes. Causes: race conditions (fix with proper waiting not timeouts), test order dependency (fix with isolated data), network variability (fix with mocking), shared state (fix with isolated browser contexts). Use the trace viewer to see exactly what happened on the failing run. Retrying is not a fix — it hides the problem. Always find the root cause.

Q10How do you structure a complete Playwright project professionally?+

Professional structure: /tests (spec files by feature), /pages (POM classes — one per page), /fixtures (custom test.extend fixtures), /test-data (JSON data files), /utils (schema validators, helpers), /auth (storageState files — gitignored), .env (credentials — gitignored), playwright.config.js (3 browser projects, baseURL, retries, reporter), global-setup.js, package.json with npm scripts for smoke/regression/report. Page classes contain only locators and actions — no test logic inside pages.

Q11What is parallel execution in Playwright and how do you configure it?+

Playwright runs tests in parallel by default using worker processes. Configure with workers: N (number of parallel workers) and fullyParallel: true (every test file runs in parallel, not just files within a spec). In CI, set workers to the number of CPU cores available. Critical constraint: tests must be fully isolated — no shared state, no dependency on execution order. If two tests write to the same database record simultaneously, you get race conditions. Fix by using unique test data per test or mocking the data layer. Parallel execution is the single fastest performance win in any large suite.

Q12What is page.evaluate() and when do you need it?+

page.evaluate() runs a JavaScript function inside the browser's JavaScript context and returns the result to your test. Use it when Playwright's built-in locators and actions cannot reach what you need: reading/writing localStorage or sessionStorage, forcing a date picker to a value by setting its .value property directly, scrolling to a specific position, checking computed CSS styles, reading values from canvas elements, or interacting with third-party widgets that don't have accessible roles. The key distinction: page.locator() finds DOM elements from the outside; page.evaluate() runs code from inside the browser. If you find yourself using evaluate() to click or fill elements, that's usually a sign to use a better locator instead.

Q13What is TypeScript and why does it matter for Playwright?+

TypeScript adds static types to JavaScript. In Playwright, this means: VS Code autocompletes every page method as you type (no more guessing spellings), type errors are caught before you run tests (if you pass a string where a number is expected, your editor tells you immediately), and POM classes are self-documenting (readonly page: Page tells anyone reading the code exactly what properties exist and what types they are). Playwright has native TypeScript support — no build step, no compilation, just rename .js to .ts and add a tsconfig.json. The migration is mostly mechanical. Most companies hiring QA automation engineers in 2024+ specify TypeScript. A JavaScript-only Playwright portfolio is functional but increasingly a yellow flag for senior reviewers.
